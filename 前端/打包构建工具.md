# 编译、打包和构建
| 工具      | 类型        | 能干             | 不能干                |
| ------- | --------- | -------------- | ------------------ |
| tsc     | 编译器       | TS→JS          | 不打包、不处理资源          |
| babel   | 编译器       | 语法降级           | 不打包                |
| swc     | 编译器       | 极速编译           | 不打包                |
| postcss | 编译器       | CSS 编译         | 不打包                |
| esbuild | 编译器 + 打包器 | 编译 + 打包        | 生态弱                |
| webpack | 打包构建工具    | 完整构建           | 编译靠 loader         |
| rollup  | 打包构建工具    | 库打包            | 复杂配置               |
| vite    | 构建工具      | 完整构建           | 底层用 esbuild+rollup |
| rspack  | 打包构建工具    | rust 版 webpack | 新生态                |
| tsup    | 构建封装工具    | 库打包            | 底层 esbuild         |
## 1. 编译（Compile）
**只做一件事：把一种代码 → 翻译成另一种代码**
- 不关心文件之间依赖关系
- 不合并文件
- 不打包
- 不处理图片、CSS、HTML
- 只管 “语法转换”
典型行为：
- TS → JS
- ESNext → ES5
- JSX → JS
- 压缩、混淆
**编译器只翻译单个文件，不做整合。**
---
## 2. 打包（Bundle）
**把一堆文件 → 整合成一个 / 几个浏览器能跑的文件**
- 处理依赖关系（import/export）
- 合并文件
- 处理资源（图片、CSS、JSON…）
- 处理路径、别名
- 代码分割、treeshaking
**打包工具 = 依赖管理者 + 文件整合者**
---
## 3. 构建（Build）

**构建 = 编译 + 打包 + 优化 + 输出最终产物**

它是**完整流程**，不是单一工具。

构建 = 编译 + 打包 + 压缩 + 代码分割 + 生成 HTML + 处理静态资源
# Babel（编译）
# TSC（编译）

Typescript Compiler，Typescript编译器，用来将单个ts文件编译成js文件，配置项都在tsconfig.json中。
## 特点
- 专注于ts编译（没有处理样式、字体...的能力），js编译，类型输出dts
- 强类型检查
- 单文件编译
## 使用场景
- 早期项目，没有复杂的编译需求
- 后端，工具库

## 缺点
- 仅仅支持ts编译
- 无法代码拆分，模块打包
- 性能差，编译大型项目十分吃力
- 配置繁琐，tsconfig，有时需要借助babel、postcss...
# Rollup
随着项目复杂度的提高，tsc完全满足不了要求，这时需要有一个接近0配置的（parcel），开箱即用的打包工具，rollup成了工具库的首选构建打包方案。

## 特点
- tree-shaking，移除未使用代码
- 简化配置
- 丰富的插件生态
## 使用场景
- UI库
- 工具库
- 按需引入的场景
## 缺点
- 大型业务项目不适合(Vite、Webpack)
- 热更新方面
- 性能方面
# Tsup
Go-->esbuild-->tsup
可以平替rollup
## 特点
- 零配置，天然支持ts
- 性能好
- 多格式
- 代码动态引入，代码分割
## 使用场景
- 工具库
- 脚手架
## 缺点
- 业务构建不推荐
- 插件生态没有rollup繁荣
