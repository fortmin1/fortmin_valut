## 1. 编译（Compile）
**只做一件事：把一种代码 → 翻译成另一种代码**
- 不关心文件之间依赖关系
- 不合并文件
- 不打包
- 不处理图片、CSS、HTML
- 只管 “语法转换”
典型行为：
- TS → JS
- ESNext → ES5
- JSX → JS
- 压缩、混淆
**编译器只翻译单个文件，不做整合。**
---
## 2. 打包（Bundle）
**把一堆文件 → 整合成一个 / 几个浏览器能跑的文件**
- 处理依赖关系（import/export）
- 合并文件
- 处理资源（图片、CSS、JSON…）
- 处理路径、别名
- 代码分割、treeshaking
**打包工具 = 依赖管理者 + 文件整合者**
---
## 3. 构建（Build）

**构建 = 编译 + 打包 + 优化 + 输出最终产物**

它是**完整流程**，不是单一工具。

构建 = 编译 + 打包 + 压缩 + 代码分割 + 生成 HTML + 处理静态资源
# Babel（编译）
# TSC（编译）

Typescript Compiler，Typescript编译器，用来将单个ts文件编译成js文件，配置项都在tsconfig.json中。
## 特点
- 专注于ts编译（没有处理样式、字体...的能力），js编译，类型输出dts
- 强类型检查
- 单文件编译
## 使用场景
- 早期项目，没有复杂的编译需求
- 后端，工具库

## 缺点
- 仅仅支持ts编译
- 无法代码拆分，模块打包
- 性能差，编译大型项目十分吃力
- 配置繁琐，tsconfig，有时需要借助babel、postcss...
# Rollup
随着项目复杂度的提高，tsc完全满足不了要求，这时需要有一个接近0配置的（parcel），开箱即用的打包工具，rollup成了工具库的首选构建打包方案。
## 特点
- tree-shaking，移除未使用代码
- 简化配置
- 丰富的插件生态
## 使用场景
- UI库
- 工具库
- 按需引入的场景
## 缺点
- 大型业务项目不适合(Vite、Webpack)
- 热更新方面
- 性能方面
# Tsup
Go-->esbuild-->tsup
可以平替rollup
## 特点
- 零配置，天然支持ts
- 性能好
- 多格式
- 代码动态引入，代码分割
## 使用场景
- 工具库
- 脚手架
## 缺点
- 业务构建不推荐
- 插件生态没有rollup繁荣
# Webpack
# Vite
整合了esbuild和rollup的现代构建工具
## 特点
- 开发环境基于esbuild
- 按需加载
- HMR
- 生成构建，rollup
## 使用场景
- 大型业务场景
- Vue、React 业务项目
## 缺点
- 生态
- 低版本兼容，legacy
- 微信网页开发、本地开发环境，es 模块
