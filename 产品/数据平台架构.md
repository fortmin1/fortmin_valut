# 一、整体架构全景（先建立直觉）

可以把它理解成一条“数据生产线”：

```
客户端埋点   ↓数据采集层（SDK / Log）   ↓消息队列（Kafka）   ↓实时/离线计算层（Flink / Spark）   ↓数据仓库（ODS → DWD → DWS → ADS）   ↓BI / 数据产品（看板 / 报表 / 指标系统）
```

---

# 二、第一层：埋点系统（Data Collection）

## 1. 本质作用

> **把用户行为“结构化”成事件数据**

---

## 2. 埋点类型

### ① 前端埋点（Web / App）

- 点击
- 曝光
- 页面访问
- 滚动
- 加购

---

### ② 后端埋点

- 下单
- 支付成功
- 登录
- 退款

---

### ③ 自动埋点 vs 手动埋点

|类型|特点|
|---|---|
|自动埋点|点击、曝光自动采集|
|手动埋点|业务事件（add_to_cart）|

---

## 3. 数据结构（核心）

所有埋点最终都是统一事件模型：

```
{  "event": "add_to_cart",  "user_id": 123,  "device_id": "abc",  "timestamp": 1710000000,  "page": "product_detail",  "properties": {    "product_id": 999,    "price": 199  }}
```

---

## 4. 埋点 SDK 做的事

- 数据收集
- 本地缓存
- 批量发送
- 重试机制
- 压缩（gzip）

---

# 三、第二层：Kafka（数据管道核心）

## 1. Kafka 在这里的角色

> **数据“缓冲 + 解耦 + 分发中心”**

---

## 2. 为什么必须用 Kafka？

如果没有 Kafka：

- 埋点 → 直接写数据库 ❌（扛不住）
- 数据丢失风险高 ❌
- 计算系统强耦合 ❌

---

## 3. Kafka 在架构中的位置

```
埋点数据   ↓Kafka Topic（事件流）   ↓↓↓↓↓↓↓↓↓↓↓↓↓Flink实时计算Spark离线计算日志存储数据仓库
```

👉 一份数据，多路消费（核心价值）

---

## 4. Kafka 的关键设计

### Topic 分类：

- user_event_topic
- order_event_topic
- payment_event_topic

---

### Partition（并行能力）

- 按 user_id hash 分区
- 保证同一用户顺序性

---

### Consumer Group

不同系统各自消费：

- 实时推荐系统
- 数据仓库ETL
- 风控系统

---

# 四、第三层：计算层（Flink / Spark）

这一层是“数据加工厂”。

---

## 1. 实时计算（Flink）

处理：

- 实时大盘（DAU）
- 实时订单监控
- 风控报警

特点：

- 秒级延迟
- 流式处理

---

## 2. 离线计算（Spark / Hive）

处理：

- 日报
- 周报
- 用户画像
- 留存分析

特点：

- T+1
- 批处理

---

## 3. 典型逻辑

```
SELECT count(distinct user_id)FROM user_eventWHERE event = 'app_open'AND date = '2026-06-29'
```

---

# 五、第四层：数据仓库（核心中的核心）

这是大厂数据体系的“中枢神经”。

---

## 1. 分层架构（经典）

```
ODS（原始数据层）   ↓DWD（明细数据层）   ↓DWS（汇总数据层）   ↓ADS（应用数据层）
```

---

## 2. 每一层作用

---

### ① ODS（原始层）

- Kafka落地数据
- 不做任何处理
- 保留原始日志

👉 类似“数据备份仓库”

---

### ② DWD（明细层）

- 清洗数据
- 去重
- 标准化字段

👉 可用但未聚合

---

### ③ DWS（汇总层）

- 用户维度聚合
- 日/周统计

例：

- 用户日活
- 用户行为次数

---

### ④ ADS（应用层）

直接给业务用：

- BI报表
- 产品看板
- KPI系统

---

## 3. 举个例子（电商）

### ODS：

```
add_to_cart 原始日志
```

### DWD：

```
清洗后的加购行为表
```

### DWS：

```
用户每日加购次数
```

### ADS：

```
加购转化率看板
```

---

# 六、第五层：BI（数据消费层）

## 1. BI 是什么？

> **把数据变成“可视化决策工具”**

---

## 2. 常见 BI 工具

- Tableau
- Power BI
- Superset（大厂常用）
- 自研数据平台

---

## 3. BI 输出内容

### ① 实时大盘

- DAU
- GMV
- 转化率

---

### ② 分析报表

- 留存分析
- 漏斗分析
- 用户画像

---

### ③ 产品看板

- 产品经理日常核心工具

---

# 七、完整数据流（端到端）

我们把全链路串起来：

```
用户行为   ↓埋点 SDK（前端/后端）   ↓Kafka（数据总线）   ↓Flink（实时计算）   ↓Spark（离线计算）   ↓数仓（ODS → DWD → DWS → ADS）   ↓BI看板 / 产品分析平台
```

---

# 八、为什么大厂一定要这样设计？

---

## 1. 解耦

埋点系统和分析系统完全分离

---

## 2. 可扩展

新增系统只需“订阅 Kafka”

---

## 3. 实时 + 离线统一

- 实时看 DAU
- 离线看留存

---

## 4. 数据一致性

所有系统消费同一数据源

---

# 九、核心难点（面试高频）

---

## 1. 数据延迟

- Kafka 堆积
- Flink 延迟

---

## 2. 数据丢失

- SDK失败
- 网络问题

---

## 3. 数据一致性

- 实时 vs 离线不一致

---

## 4. 指标口径统一

👉 最大坑：

> “同一个DAU，不同部门算出来不一样”

---

# 十、一句话总结（非常重要）

---

> **大厂数据平台 = 用埋点采集行为数据 → Kafka解耦传输 → Flink/Spark加工 → 数仓分层建模 → BI驱动决策**
