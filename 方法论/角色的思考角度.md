# 角色的思考角度

我以前分析软件系统时，容易直接从“功能、架构、技术、运维、业务”这些维度切入。

这些维度本身没有问题，但遇到一个具体产品时还是会混乱，原因是：

> **没有先确定自己当前站在哪个角色上。**

同一个系统，不同角色看到的不是同一件事。

例如北森：

- 用户看到：能不能方便地招聘、考勤、绩效管理
- 产品经理看到：目标用户是谁、核心场景是什么、指标怎么增长
- 项目经理看到：需求怎么排期、资源怎么协调、风险怎么控制
- 开发工程师看到：模块怎么实现、接口怎么设计、数据库怎么建
- 架构师看到：系统如何分层、如何扩展、如何保证安全和稳定
- CTO看到：技术路线、团队能力、成本、长期演进
- CEO看到：商业模式、市场空间、组织效率、竞争壁垒

所以分析一个软件系统时，可以先问：

> **我现在是以什么角色在看这个系统？**

角色确定之后，问题才会变清晰。

---

# 一、为什么要先确定角色？

因为角色决定三件事：

## 1. 关注目标不同

产品经理关心：

```
用户价值
业务指标
需求优先级
```

架构师关心：

```
系统结构
质量属性
技术取舍
```

运维关心：

```
可用性
监控告警
发布回滚
```

CEO关心：

```
收入
成本
增长
壁垒
```

---

## 2. 判断标准不同

同一个功能，不同角色判断好坏的标准不同。

例如“异地登录验证码”。

用户视角：

```
会不会打扰我？
验证流程麻不麻烦？
```

产品视角：

```
是否降低盗号风险？
是否影响登录转化率？
```

安全视角：

```
风险识别是否准确？
是否能防止撞库、盗号、异常设备登录？
```

架构视角：

```
风控规则如何抽象？
验证码服务是否可复用？
是否会影响登录链路性能？
```

运维视角：

```
短信服务挂了怎么办？
验证码接口被刷怎么办？
告警如何配置？
```

---

## 3. 输出结果不同

产品经理最后输出：

```
需求文档
指标体系
版本规划
```

架构师最后输出：

```
架构图
模块划分
技术选型
风险清单
```

项目经理最后输出：

```
排期计划
资源安排
风险跟踪
```

CTO最后输出：

```
技术战略
团队规划
成本控制
长期演进路线
```

所以角色不是“头衔”，而是一套思考任务。

---

# 二、分析软件系统的角色地图

可以把角色分成 5 层。

```
市场与战略层：CEO / 创始人 / 业务负责人
产品与增长层：产品经理 / 运营 / 数据分析
交付与协作层：项目经理 / 研发负责人 / 测试
技术与架构层：开发工程师 / 架构师 / CTO
运行与治理层：运维 / 安全 / 合规 / 客服
```

这 5 层不是互相割裂的。

一个成熟的软件系统，通常要同时被这些角色理解。

# 三、用户视角：这个产品好不好用？

用户视角是所有视角的起点。

核心问题：

```
用户是谁？
用户在什么场景下使用？
用户想完成什么任务？
当前流程哪里痛苦？
这个产品有没有让事情变简单？
```

关注点：

- 使用场景
- 操作路径
- 学习成本
- 响应速度
- 稳定性
- 体验细节

分析模板：

```
目标用户：
核心场景：
用户任务：
原来的做法：
现在的做法：
减少了什么成本：
增加了什么价值：
```

例如医院急救物资系统：

```
目标用户：护士、库房管理员
核心场景：急救物资盘点、出入库、效期检查
用户任务：快速知道物资是否齐全、是否过期、是否需要补充
原来的做法：人工盘点、纸质登记、Excel记录
现在的做法：RFID识别、系统自动更新库存、异常提醒
减少成本：人工盘点时间、漏盘风险、过期风险
增加价值：物资可追溯、库存更准确、急救准备更可靠
```

# 四、产品经理视角：这个产品解决什么问题？

产品经理关心的是：

> **用户价值如何转化为产品方案和业务指标。**

核心问题：

```
目标用户是谁？
用户痛点是什么？
核心场景是什么？
产品提供什么解决方案？
需求优先级怎么排？
成功指标是什么？
```

关注点：

- 用户分层
- 使用场景
- 需求价值
- 功能边界
- 版本规划
- 数据指标
- 转化漏斗

常见输出：

- PRD
- 用户流程图
- 功能列表
- 原型图
- 指标体系
- 版本路线图

产品经理分析一个功能时，可以问：

```
1. 这个功能服务哪个用户？
2. 用户为什么需要它？
3. 没有这个功能会怎样？
4. 它影响哪个核心指标？
5. 它是核心需求、增值需求，还是边缘需求？
6. 它的最小可用版本是什么？
7. 如何验证它真的有价值？
```

例如“数据看板”：

```
用户：管理者
痛点：不知道物资消耗趋势和异常情况
方案：库存、消耗、预警、过期物资可视化
指标：盘点耗时下降、过期物资数量下降、补货及时率提升
```

可以关联：

- [[产品/设计指标体系]]
- [[产品/数据驱动决策]]
- [[产品/漏斗分析+AB决策]]

# 五、项目经理视角：这件事怎么落地？

项目经理关心的是：

> **在有限时间、有限资源、有限人力下，把事情交付出来。**

核心问题：

```
目标是什么？
范围是什么？
谁来做？
什么时候完成？
风险在哪里？
依赖谁？
如何验收？
```

关注点：

- 需求范围
- 排期
- 人力
- 里程碑
- 风险
- 沟通
- 验收标准

常见输出：

- 项目计划
- 任务拆解
- 里程碑
- 风险清单
- 周报
- 验收清单

项目经理分析一个需求时，可以问：

```
1. 这个需求的交付范围是什么？
2. 哪些必须做，哪些可以后做？
3. 前端、后端、测试、运维分别需要做什么？
4. 是否依赖第三方系统、硬件、接口、数据？
5. 最大风险是什么？
6. 如果延期，砍什么范围？
7. 交付完成的标准是什么？
```

例如 RFID 盘点功能：

```
范围：RFID设备读取、库存匹配、盘点记录、异常提示
依赖：RFID硬件、设备协议、测试环境、真实物资标签
风险：设备识别率不足、现场网络不稳定、数据和实物不一致
验收：能完成一次完整盘点，并生成差异记录
```

# 六、开发工程师视角：这个功能怎么实现？

开发工程师关心的是：

> **把需求变成可运行、可维护、可测试的代码。**

核心问题：

```
模块怎么拆？
接口怎么设计？
数据怎么存？
异常怎么处理？
权限怎么控制？
代码如何复用？
测试怎么覆盖？
```

关注点：

- 功能模块
- API设计
- 数据模型
- 业务规则
- 权限校验
- 异常处理
- 性能问题
- 单元测试和集成测试

常见输出：

- 接口文档
- 数据库表设计
- 代码实现
- 单元测试
- 技术文档

开发工程师分析一个功能时，可以问：

```
1. 输入是什么？
2. 输出是什么？
3. 核心业务规则是什么？
4. 数据库表怎么设计？
5. 是否需要事务？
6. 是否需要缓存？
7. 是否有并发问题？
8. 失败时如何回滚或补偿？
9. 权限如何校验？
10. 如何测试？
```

例如“物资出库”：

```
输入：物资ID、数量、领用人、科室、用途
输出：出库记录、库存扣减结果
规则：库存不能扣成负数；高价值物资可能需要审批
数据：material、inventory、outbound_record
事务：出库记录生成和库存扣减必须同时成功
权限：护士可申请，管理员可审核
测试：正常出库、库存不足、重复提交、无权限操作
```

# 七、架构师视角：系统如何组织？

架构师关心的是：

> **系统结构如何支撑业务长期发展。**

开发工程师偏具体实现，架构师偏整体组织。

核心问题：

```
业务边界是什么？
系统应该分成哪些模块？
核心质量属性是什么？
哪些地方会成为瓶颈？
哪些能力需要复用？
技术选型是否匹配业务？
未来如何扩展？
```

关注点：

- 业务领域
- 模块边界
- 分层设计
- 质量属性
- 数据流
- 集成方式
- 技术选型
- 可扩展性
- 可维护性

常见输出：

- 系统架构图
- 分层架构图
- 模块关系图
- 数据流图
- 技术选型说明
- 架构风险清单

架构师分析系统时，可以问：

```
1. 这个系统属于什么业务领域？
2. 核心业务流程是什么？
3. 最重要的质量属性是什么？
4. 系统有哪些核心模块？
5. 哪些模块是通用能力？
6. 数据如何流转？
7. 和外部系统如何集成？
8. 当前架构能支撑多大规模？
9. 未来增长后哪里会先出问题？
10. 哪些技术是必要的，哪些是过度设计？
```

可以结合 [[方法论/软件分层视角]]：

```
业务层
应用功能层
安全层
数据层
基础设施层
运维层
架构层
治理层
```

也可以结合 [[方法论/业务驱动架构]]：

```
业务领域
↓
核心用户
↓
核心场景
↓
质量属性
↓
系统架构
↓
技术选型
↓
部署架构
↓
运维体系
```

# 八、CTO视角：技术如何支撑公司？

CTO不是“大号架构师”。

架构师更多关注系统，CTO还要关注：

```
技术战略
团队能力
研发效率
成本
风险
长期演进
```

核心问题：

```
技术路线是否支持业务战略？
团队能不能交付这个系统？
当前技术债会不会拖垮后续发展？
研发效率如何提升？
基础设施成本是否可控？
安全和稳定性风险是否可接受？
技术是否能形成壁垒？
```

关注点：

- 技术战略
- 技术选型
- 团队组织
- 研发流程
- 工程效率
- 技术债
- 成本控制
- 安全稳定
- 平台化能力

常见输出：

- 技术路线图
- 团队规划
- 研发流程规范
- 平台化建设方案
- 成本优化方案
- 技术风险评估

CTO分析系统时，可以问：

```
1. 这套技术能支撑未来 1-3 年业务吗？
2. 团队是否掌握这些技术？
3. 关键系统是否依赖少数人？
4. 哪些能力应该平台化？
5. 哪些地方需要自研，哪些应该采购？
6. 成本主要花在哪里？
7. 技术债在哪里？
8. 最大稳定性风险是什么？
9. 最大安全风险是什么？
10. 技术能否成为业务壁垒？
```

例如企业 SaaS：

```
短期：快速交付客户需求
中期：沉淀权限、工作流、报表、消息等通用能力
长期：平台化、多租户、数据分析、AI能力
```

# 九、运维 / SRE 视角：系统如何稳定运行？

运维关心的是：

> **系统上线之后能不能持续稳定地跑。**

核心问题：

```
系统部署在哪里？
如何发布？
如何监控？
出问题怎么发现？
故障怎么恢复？
数据怎么备份？
容量够不够？
```

关注点：

- 部署架构
- CI/CD
- 监控
- 日志
- 告警
- 备份
- 容灾
- 扩容
- 故障演练

常见输出：

- 部署文档
- 运维手册
- 监控面板
- 告警规则
- 应急预案
- 备份策略

运维分析系统时，可以问：

```
1. 服务部署在哪些机器或容器？
2. 依赖哪些中间件？
3. 哪些指标必须监控？
4. 日志是否能定位问题？
5. 发布失败如何回滚？
6. 数据库如何备份和恢复？
7. 单点故障在哪里？
8. 流量突然上涨怎么办？
9. 第三方服务不可用怎么办？
10. 有没有应急预案？
```

例如验证码服务：

```
监控：验证码发送成功率、短信接口耗时、失败率
告警：短信发送失败率超过阈值
降级：短信服务异常时切换备用通道
风控：限制单个手机号和IP的发送频率
```

# 十、安全 / 合规视角：系统有什么风险？

安全角色关心的是：

> **谁能访问什么，数据会不会泄露，行为能不能追溯。**

核心问题：

```
身份如何认证？
权限如何控制？
敏感数据如何保护？
异常行为如何识别？
关键操作是否可审计？
是否满足合规要求？
```

关注点：

- 认证
- 授权
- 数据权限
- 风控
- 审计
- 加密
- 隐私
- 合规

常见输出：

- 权限模型
- 安全方案
- 风控规则
- 审计日志设计
- 数据脱敏方案
- 合规检查清单

安全分析系统时，可以问：

```
1. 用户如何登录？
2. 权限模型是什么？
3. 是否有越权风险？
4. 敏感数据有哪些？
5. 数据传输和存储是否加密？
6. 关键操作是否记录审计日志？
7. 异常登录、异常操作如何处理？
8. 是否有接口刷量、撞库、爬虫风险？
9. 数据导出是否受控？
10. 是否满足行业合规要求？
```

例如 HR 系统：

```
敏感数据：简历、身份证、手机号、薪资、绩效
权限风险：普通员工看到他人薪资
审计要求：谁查看过候选人信息，谁修改过绩效记录
风控场景：异地登录、大量导出简历、频繁失败登录
```

# 十一、数据分析视角：如何证明产品有效？

数据分析关心的是：

> **用数据判断业务是否变好。**

核心问题：

```
核心指标是什么？
用户行为如何记录？
漏斗在哪里流失？
功能上线后有没有效果？
异常数据说明什么问题？
```

关注点：

- 指标体系
- 埋点
- 漏斗
- 留存
- 转化
- A/B测试
- 数据看板

常见输出：

- 指标体系
- 埋点方案
- 数据看板
- 分析报告
- 实验结论

数据分析系统时，可以问：

```
1. 北极星指标是什么？
2. 过程指标有哪些？
3. 每个关键行为是否可记录？
4. 用户在哪一步流失？
5. 不同用户群体表现是否不同？
6. 新功能上线前后指标如何变化？
7. 是否需要 A/B 测试？
8. 数据是否准确？
9. 指标是否被错误解读？
10. 数据结论能指导什么行动？
```

例如招聘系统：

```
核心指标：招聘成功率、招聘周期
过程指标：简历投递数、筛选通过率、面试邀约率、offer接受率
问题定位：如果简历多但面试少，可能是筛选环节或岗位描述有问题
```

可以关联：

- [[产品/数据平台架构]]
- [[产品/设计指标体系]]

# 十二、业务负责人 / CEO视角：这个系统是否值得做？

业务负责人关心的是：

> **这个产品能不能创造商业价值。**

核心问题：

```
市场是否存在？
客户是否愿意付费？
商业模式是什么？
竞争壁垒是什么？
成本结构是否健康？
增长路径是什么？
```

关注点：

- 市场空间
- 客户价值
- 收入模型
- 成本结构
- 增长策略
- 竞争格局
- 商业壁垒

常见输出：

- 商业计划
- 市场分析
- 收入预测
- 成本测算
- 增长策略
- 竞争分析

CEO分析产品时，可以问：

```
1. 这个产品解决的是不是高价值问题？
2. 谁付钱？
3. 为什么现在是做它的好时机？
4. 市场有多大？
5. 竞争对手是谁？
6. 我们的差异化在哪里？
7. 获客成本是否可控？
8. 毛利是否健康？
9. 能不能规模化复制？
10. 壁垒来自哪里？
```

例如企业 SaaS：

```
客户：企业
付费方：老板、部门负责人、采购部门
价值：提高组织效率、降低管理成本、减少错误
成本：研发、销售、实施、客服、云资源
壁垒：行业Know-how、客户数据、集成能力、服务能力、品牌
```

# 十三、不同角色之间如何连接？

角色不是孤立的。

一个需求从想法到上线，大概会经过这条链路：

```
CEO / 业务负责人
↓
产品经理
↓
项目经理
↓
架构师
↓
开发工程师
↓
测试
↓
运维
↓
数据分析
↓
继续反馈给产品和业务
```

可以理解为：

```
商业目标
↓
用户问题
↓
产品方案
↓
项目计划
↓
技术架构
↓
代码实现
↓
上线运行
↓
数据验证
↓
业务迭代
```

这和 [[方法论/业务驱动架构]] 是一致的。

区别是：

- 业务驱动架构：从系统维度看
- 角色思考角度：从人和职责维度看

# 十四、分析具体产品时的推荐顺序

如果只是学习一个软件系统，建议不要一上来就问技术。

可以按这个顺序：

## 第一步：先站在用户视角

```
谁在用？
在什么场景用？
解决什么问题？
```

## 第二步：切到产品经理视角

```
核心功能是什么？
需求优先级是什么？
指标是什么？
```

## 第三步：切到业务负责人视角

```
为什么这个产品值得做？
商业价值在哪里？
```

## 第四步：切到架构师视角

```
系统怎么分层？
核心模块是什么？
质量属性是什么？
```

## 第五步：切到开发工程师视角

```
功能如何实现？
接口、数据库、事务、缓存怎么设计？
```

## 第六步：切到运维和安全视角

```
系统如何稳定运行？
数据和权限如何保护？
```

## 第七步：切到数据分析视角

```
上线后如何判断它有效？
下一步怎么优化？
```

最终形成：

```
用户价值
↓
产品方案
↓
商业价值
↓
技术架构
↓
工程实现
↓
稳定运行
↓
数据反馈
```

# 十六、用这个框架分析医院急救物资系统

## 用户视角

```
用户：护士、库房管理员
场景：急救物资盘点、出入库、效期检查
问题：人工盘点慢、容易漏盘、过期物资不容易发现
价值：自动识别、及时预警、减少人工错误
```

## 产品经理视角

```
核心功能：物资管理、RFID盘点、出入库、效期预警、报表
优先级：盘点准确性 > 出入库流程 > 预警 > 统计分析
指标：盘点耗时、库存准确率、过期物资数量、补货及时率
```

## 项目经理视角

```
范围：Web系统 + 后台接口 + 数据库 + RFID设备接入
依赖：RFID读写器、标签、测试物资、现场网络
风险：硬件识别率、设备协议、真实场景数据不完整
验收：完成一次真实盘点并生成库存差异结果
```

## 架构师视角

```
业务领域：医疗 + IoT + 资产管理
质量属性：准确性 > 可追溯性 > 易用性 > 稳定性
核心模块：用户权限、物资档案、库存、盘点、出入库、预警、报表
数据流：RFID设备 → 后台服务 → 库存数据 → 预警/报表
技术选型：Spring Boot + MySQL + Redis + Nginx + Docker
```

## 开发工程师视角

```
表设计：material、inventory、rfid_tag、check_task、check_record、inbound_record、outbound_record
事务：出入库记录和库存变更必须一致
权限：护士、管理员、系统管理员
异常：重复扫描、未知标签、库存不足、设备离线
测试：正常盘点、漏盘、多扫、重复提交、无权限操作
```

## 运维视角

```
监控：服务状态、数据库连接、接口耗时、设备在线状态
日志：RFID读取日志、库存变更日志、用户操作日志
备份：数据库定时备份
故障：设备不可用时支持人工录入或补录
```

## 安全视角

```
认证：账号密码 / JWT
权限：RBAC
敏感数据：人员信息、操作记录、物资流转记录
审计：谁领取、谁归还、谁修改库存、谁处理预警
风险：越权修改库存、删除记录、伪造出库
```

## 数据分析视角

```
北极星指标：急救物资可用率
过程指标：盘点完成率、库存准确率、预警处理时长、物资过期率
看板：库存总览、缺货预警、近效期物资、科室领用排行
优化：根据消耗趋势做补货建议
```

# 十七、角色切换的核心原则

以后分析软件系统时，不要混在一起想。

可以这样提醒自己：

```
我现在是用户：先问好不好用
我现在是产品经理：先问有没有价值
我现在是项目经理：先问怎么交付
我现在是开发：先问怎么实现
我现在是架构师：先问怎么组织
我现在是CTO：先问长期是否支撑业务
我现在是运维：先问怎么稳定运行
我现在是安全：先问风险在哪里
我现在是数据分析：先问如何证明有效
我现在是CEO：先问是否值得做
```

最重要的一句话：

> **角色决定问题，问题决定视角，视角决定答案。**

如果感觉混乱，通常不是知识不够，而是同时开了太多角色。

先选一个角色，再开始分析。
