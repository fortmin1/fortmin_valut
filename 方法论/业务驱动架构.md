# 第一部分：软件可以按什么分类？

如果站在最高层次看，软件首先是按**业务领域（Domain）**分类的。

## 1. 企业管理系统（MIS/ERP类）

例如：

- 北森
- OA
- ERP
- CRM
- WMS
- MES

特点：

- 数据录入多
- 表单多
- 审批多
- 权限复杂

技术重点：

- RBAC权限
- 工作流
- 数据一致性
- 报表

---

## 2. 电商系统

例如：

- 淘宝
- 京东
- 拼多多

特点：

- 用户量大
- 并发高
- 活动流量波动大

技术重点：

- 缓存
- 分布式
- 高并发
- 秒杀

---

## 3. 社交系统

例如：

- 微信
- QQ
- Discord

特点：

- 实时通信

技术重点：

- WebSocket
- IM消息
- MQ
- 在线状态

---

## 4. 内容平台

例如：

- 抖音
- B站
- YouTube

特点：

- 视频巨大

技术重点：

- CDN
- 对象存储
- 视频转码
- 推荐算法

---

## 5. 金融系统

例如：

- 银行
- 支付宝
- 证券

特点：

- 绝对不能算错钱

技术重点：

- 事务
- 审计
- 风控
- 高可用

---

## 6. 工业控制系统（ICS）

例如：

- 配电系统
- 电网调度
- 工厂MES
- PLC控制

特点：

- 设备比人重要

技术重点：

- 实时性
- 稳定性
- 安全隔离

---

## 7. 游戏系统

例如：

- 王者荣耀
- 原神

特点：

- 延迟敏感

技术重点：

- 实时同步
- 状态同步
- 网络优化

---

## 8. 物联网系统（IoT）

例如：

- 智能家居
- RFID系统
- 智能急救箱

你的毕业设计实际上属于：

> 医疗 + IoT + 资产管理系统

技术重点：

- 设备接入
- MQTT
- RFID
- 边缘计算

---

# 第二部分：业务会不会影响其它层？

答案是：

> **业务决定一切。**

架构师有一句经典的话：

> Architecture follows business.

（架构服务于业务）

---

# 举个例子

同样是“登录”。

## 北森

登录失败：

```
用户名密码↓短信验证码↓登录成功
```

允许：

- 2秒
- 5秒

没人介意。

---

## 王者荣耀

登录耗时：

```
5秒
```

用户：

```
垃圾游戏
```

卸载。

---

同一个功能：

业务不同

↓

架构要求完全不同。

---

# 再看数据库

## 医院物资系统

100个护士

同时在线：

```
50人
```

MySQL：

```
单机即可
```

---

## 淘宝

双十一：

```
100万用户
```

同时下单：

```
10万TPS
```

单机数据库直接崩。

必须：

- 分库分表
- Redis
- MQ

---

# 第三部分：一个业务会逐层影响整个系统

可以理解成：

```
业务 ↓功能 ↓非功能需求 ↓技术架构 ↓基础设施
```

例如：

---

配电系统

↓

要求：

```
7×24运行
```

↓

非功能需求：

```
高可靠
```

↓

架构：

```
主备双活集群
```

↓

基础设施：

```
双机房UPS柴油发电机
```

---

抖音

↓

要求：

```
看视频不卡
```

↓

非功能需求：

```
高带宽低延迟
```

↓

架构：

```
CDN视频转码推荐系统
```

↓

基础设施：

```
大量GPU大量存储
```

# 第四部分：架构师最常用的思考框架

看到一个系统时，先问：

## 1. 核心业务是什么？

例如：

- 电商
- 金融
- 游戏
- 工控

---

## 2. 最重要的质量属性是什么？

质量属性（Quality Attributes）比功能更重要。

常见有：

|属性|说明|
|---|---|
|性能|快|
|可用性|不宕机|
|安全性|不被攻击|
|可扩展性|能扩容|
|可维护性|易开发|
|实时性|延迟低|
|一致性|数据准确|

---

例如：

### 银行

优先级：

```
一致性>安全性>性能
```

---

### 游戏

优先级：

```
实时性>性能>一致性
```

---

### 配电系统

优先级：

```
可靠性>实时性>安全性
```

---

### 北森

优先级：

```
安全性>可维护性>功能完整性
```

---

# 最终形成一个完整视角

以后看到任何系统，可以先画这样一个分析模型：

```
业务领域    ↓核心用户    ↓核心场景    ↓质量属性    ↓系统架构    ↓技术选型    ↓部署架构    ↓运维体系
```

例如你的急救物资系统：

```
业务领域：医院物资管理核心用户：护士、管理员核心场景：RFID盘点质量属性：准确性 > 可追溯性 > 易用性系统架构：SpringBoot + Vue技术选型：MySQL + Redis + RFID部署架构：Nginx + Docker运维体系：日志 + 备份
```

这样分析时，你看到的就不再是“一个软件”，而是一个从业务目标一直延伸到底层服务器和运维体系的完整系统。
